Lab 1: ArchUnit

  1. Open the architectural-design-patterns project in your favorite editor or IDE

  2. In the src/test/java directory, and in the com.jitterted.ebp.blackjack package create a file called HexArchTest.java

  3. In the file, copy the following content

    package com.jitterted.ebp.blackjack;

import com.tngtech.archunit.core.domain.JavaClasses;
import com.tngtech.archunit.core.importer.ClassFileImporter;
import com.tngtech.archunit.lang.ArchRule;
import org.junit.jupiter.api.Test;

import static com.tngtech.archunit.lang.syntax.ArchRuleDefinition.noClasses;

public class HexArchTest {

  @Test
  public void domainMustNotDependOnAdapter() throws Exception {
    JavaClasses importedClasses =
        new ClassFileImporter().importPackages("com.jitterted.ebp.blackjack");

    ArchRule myRule = noClasses().that().resideInAPackage("..domain..")
                                 .should().dependOnClassesThat()
                                 .resideInAPackage("..adapter..");

    myRule.check(importedClasses);
  }
}

  4. Run the test in either your editor, or run it using mvn test in the command line.

  5. Does it violate our architectural rule?

Lab 5: Competing Consumers

This will not work on Apple Silicon M1, unfortunately because the amd64 will not run. While docker can emulate for the most, it will not in this case.

  1. Open a terminal and navigate to your architectural-design-patterns project and into the competing-consumers folder

  2. Run the command docker-compose up -d broker zookeeper control-center

  3. Wait until all the components are loaded that you can monitor with docker ps

  4. In your browser of choice, open http://localhost:9021 and click on the broker labeled controlcenter-cluster

  5. Next, click on the Topics section on the left menu

  6. Click on Add Topic button on the upper right hand corner

  7. In this new topic window, name the new topic my-orders and enter 3 partitions

  8. Click the Create with Defaults button

  9. Go back to your terminal and run docker-compose up -d my-producer my-consumer-1 my-consumer-2 my-consumer-3

  10. View the logs of the running consumers, for example, to view my-consumer-3 we can use docker logs my-consumer-3 or docker logs my-consumer-3 -f and view the distribution of orders

  11. Knock one of the consumers off using docker-compose stop my-consumer-3 and view the logs of both my-consumer-2 and my-consumer-1

  12. What do the logs say?

    In the logs, look for Partitions Revoked and Partitions Assigned. What you are looking for is a consumer picking up the slack of another consumer.

  13. Run docker-compose down in the competing-consumers folder

Lab 6: Claim Check

This will not work on Apple Silicon M1, unfortunately because the amd64 will not run. While docker can emulate for the most, it will not in this case.

  1. Open a terminal and navigate to your architectural-design-patterns project and into the claim_cqrs folder

  2. Run the command docker-compose up -d broker zookeeper control-center

  3. Wait until all the components are loaded that you can monitor with docker ps

  4. In your browser of choice, open http://localhost:9021 and click on the broker labeled controlcenter-cluster

  5. Next, click on the Topics section on the left menu

  6. Click on Add Topic button on the upper right hand corner

  7. In this new topic window, name the new topic my-avro-orders and enter 3 partitions

  8. Click the Create with Defaults button

  9. Go back to your terminal and run docker-compose up -d my-avro-producer my-avro-consumer-1 my-avro-consumer-2 my-avro-consumer-3

  10. Visit http://localhost:8081/subjects and what do you see? Note one of the subjects, my-avro-orders-value

  11. Visit http://localhost:8081/subjects/my-avro-orders-value/versions and what do you see? Note the version number

  12. Visit http://localhost:8081/subjects/my-avro-orders-value/versions/{versionId} where you will replace {version} with the version you noted in the previous step. What do you see?

  13. The idea here is that there a schema involved and you are looking at the storage, the claim check is the id you see in this payload. This schema is not sent with the message

  14. Keep this instances up in docker for the next lab

Lab 7: CQRS

  1. In your browser of choice, open http://localhost:9021 and click on the broker labeled controlcenter-cluster

  2. Select on the left hand menu ksqlDB

  3. Select ksqldb1 instance

  4. In the Editor, enter the following:

    CREATE STREAM my_avro_orders \
(total BIGINT, shipping VARCHAR, state VARCHAR, discount DOUBLE, \
gender VARCHAR) WITH (kafka_topic='my-avro-orders', value_format='AVRO');

  5. Select auto.offset.reset to earliest

  6. In your editor, enter the following to create non-persistent query using SELECT * from my_avro_orders emit changes; and review the results

  7. In your editor, enter the following to create another non-persistent query using SELECT state, shipping from my_avro_orders emit changes; and review the results

  8. In your editor, enter the following to create another non-persistent query using SELECT state, count(*) as count from my_avro_orders group by state emit changes; and review the results

  9. In your editor, enter the following to create another persistent query using CREATE TABLE my_avro_state_count AS SELECT state, count(*) as count from my_avro_orders group by state emit changes;

  10. Click on Topics on the left hand menu, notice the new topic MY_AVRO_STATE_COUNT, view the messages, this is a materialized view that we can port over to a database.

  11. When done, in the terminal, type docker compose down
