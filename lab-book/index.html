Lab 1: ArchUnit

  1. Open the architectural-design-patterns project in GitPod

  2. In the hex-arch module, and in the src/test/java directory, and in the com.jitterted.ebp.blackjack package create a file called HexArchTest.java

  3. In the file, copy the following content

    package com.jitterted.ebp.blackjack;

import com.tngtech.archunit.core.domain.JavaClasses;
import com.tngtech.archunit.core.importer.ClassFileImporter;
import com.tngtech.archunit.lang.ArchRule;
import org.junit.jupiter.api.Test;

import static com.tngtech.archunit.lang.syntax.ArchRuleDefinition.noClasses;

public class HexArchTest {

  @Test
  public void domainMustNotDependOnAdapter() throws Exception {
    JavaClasses importedClasses =
        new ClassFileImporter().importPackages("com.jitterted.ebp.blackjack");

    ArchRule myRule = noClasses().that().resideInAPackage("..domain..")
                                 .should().dependOnClassesThat()
                                 .resideInAPackage("..adapter..");

    myRule.check(importedClasses);
  }
}

  4. Run the test in either your editor, or run it using mvn test in the command line.

  5. Does it violate our architectural rule?

Lab 5: Competing Consumers

  1. Open the competing-consumers module folder

  2. Right-click on the docker-compose.yml file and select "Compose Up - Select Services", deselect all the checkmarks, and select control-center

  3. Wait until all the components are loaded that you can monitor with docker ps

  4. In your browser of choice, open port 9021 in your gitpod ports menu

  5. Next, click on the Topics section on the left menu

  6. Click on Add Topic button on the upper right hand corner

  7. In this new topic window, name the new topic my-orders and enter 3 partitions

  8. Click the Create with Defaults button

  9. Go back to the competing-consumers module folder

  10. Right-click on the docker-compose.yml file and select "Compose Up - Select Services", deselect all the checkmarks, and just select my-producer, my-consumer-1, my-consumer-2, my-consumer-3

  11. View the logs of the running consumers by right-clicking on the container in the Docker menu and selecting "View Logs"

  12. Knock one of the consumers off by right-clicking one of the consumer containers, like my-consumer-3 and select and view the logs of both my-consumer-2 and my-consumer-1

  13. What do the logs say?

    In the logs, look for Partitions Revoked and Partitions Assigned. What you are looking for is a consumer picking up the slack of another consumer.

  14. Run docker-compose down in the competing-consumers folder, by selecting the docker_compose.yml file, right-clicking, and selecting "Compose Down"

  9. Navigate to your architectural-design-patterns project and into the claim-check module once again.

  10. Right-click on the docker-compose.yml file and select "Compose Up - Select Services", deselect all the checkmarks, and select my-avro-producer, my-avro-consumer-1, my-avro-consumer-2, my-avro-consumer-3

  11. Open port 8081, from the ports menu, Visit http://<url>:8081/subjects and what do you see? Note one of the subjects, my-avro-orders-value

  12. Visit http://<url>:8081/subjects/my-avro-orders-value/versions and what do you see? Note the version number

  13. Visit http://<url>:8081/subjects/my-avro-orders-value/versions/{versionId} where you will replace {versionId} with the version you noted in the previous step. What do you see?

  14. The idea here is that there a schema involved, and you are looking at the storage, the claim check is the id you see in this payload. This schema is not sent with the message

  15. Run docker-compose down in the claim-check module folder, by selecting the docker_compose.yml file, right-clicking, and selecting "Compose Down"

Lab 7: CQRS (Command Query Responsibility Segregation)

  1. Open the cqrs module folder

  2. Right-click on the docker-compose.yml file and select "Compose Up - Select Services", deselect all the checkmarks, and select connect ksqldb-cli postgres control-center

  3. Login into connect container by using either Attach Shell on Gitpod or docker exec -it connect /bin/bash

  4. Run the following in the container confluent-hub install mongodb/kafka-connect-mongodb:1.10.0, or whatever the latest version is from Confluent Hub

  5. Select 2. / (where this tool is installed)

  6. Answer y to Do you want to install this into /usr/share/confluent-hub-components?

  7. Answer y to I agree to the software license agreement (yN)

  8. Answer y to Do you want to continue?

  9. Answer y to Do you want to update all detected configs? (yN)

  10. Go to KsqlDB to create Stream

    CREATE STREAM stock_trades WITH (
KAFKA_TOPIC = 'postgres_stock_trade',
VALUE_FORMAT = 'AVRO'
);

  11. Go to KSQL-CLI Container by either attaching to the shell using docker exec ksqldb-cli /bin/bash

  12. Run following commands

    1. ksql http://ksqldb-server:8088

    2. show streams;

    3. SET 'auto.offset.reset'='earliest';

    4. select * from STOCK_TRADES;

    5. select stock_symbol,count() as count from STOCK_TRADES group by STOCK_SYMBOL emit changes; 12.6 select stock_symbol,count() as count, topk(stock_symbol, 5) from STOCK_TRADES group by STOCK_SYMBOL emit changes;

    6. create an aggregate topic - create table stock_count with (PARTITIONS = 3, VALUE_FORMAT = 'JSON') as select STOCK_SYMBOL, count(*) as count from STOCK_TRADES group by stock_symbol EMIT CHANGES;

  13. Setup MongoDB sink (read data from Aggregate Topic and push data to MongoDB)

  14. Configure MongoDB sink using mongosink.json that is in src\main\resources

  15. Use mongodb-express container to see the results for the database

  16. Run the following in the container a MongoDB Connect that reads from postgres and do the rest as per Outbox pattern - confluent-hub install confluentinc/kafka-connect-jdbc:10.7.1

  17. Select 2. / (where this tool is installed)

  18. Answer y to Do you want to install this into /usr/share/confluent-hub-components?

  19. Answer y to I agree to the software license agreement (yN)

  20. Answer y to Do you want to continue?

  21. Answer y to Do you want to update all detected configs? (yN)

  22. Exit the container using exit

  23. Restart the container using docker restart connect

  24. In your application mvn exec:java -Dexec.mainClass=com.xyzcorp.outbox.CreateStocks to generate data. You can also run in your IDE

  25. Login into your postgres container using docker exec -it postgres /bin/bash

  26. Run the following: export PGPASSWORD='docker'

  27. Run the following: psql -d docker -U docker

  28. In the postgres shell run \dt which will show all the tables

  29. In the postgres shell run \d stock_trade, which will show specific table schema

  30. Run SELECT * from stock_trade; and ensure that the data exists

  31. Setup the JDBC connector (reads data from source table and create Kafka topics)

  32. Key Converter Class - io.confluent.connect.avro.AvroConverter

  33. Value Converter Class - io.confluent.connect.avro.AvroConverter

  34. Database Connection and JDBC URL jdbc:postgresql://postgres:5432/

  35. Database JDBC User & Pass use docker

  36. Database Dialect PostgressSqlDatabaseDialect

  37. Database Mode incrementing

  38. Topic Prefix - postgres`_

  39. Additional Properties key.converter.schema.registry.url set to http://schema-registry:8081

  40. Additional Properties → value.converter.schema.registry.url set to http://schema-registry:8081

  41. Go to KsqlDB to create Stream

KAFKA_TOPIC = 'postgres_stock_trade',

  42. Go to KSQL-CLI Container by either attaching to the shell using docker exec ksqldb-cli /bin/bash

    6. create an aggregate topic - create table stock_count with (PARTITIONS = 3, VALUE_FORMAT = 'JSON') as select STOCK_SYMBOL, count(*) as count from STOCK_TRADES group by stock_symbol EMIT CHANGES;
